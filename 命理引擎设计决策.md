## 设计哲学

整个命理引擎遵循一条主线：

> **排盘输入 → 内容寻址生成不可变 ChartSnapshot → 确定性算好喜忌和神煞 → LLM 只做润色不做判断**

这样命理引擎是确定的、可缓存的、可复现的；LLM 只在最外层负责"说人话"。所有可能有流派争议的判断（用神、格局分类）要么规则定死（喜忌用扶抑+调候），要么砍掉（格局判定），绝不交给 LLM 现场判。

## 决策 1：喜忌 = 确定性模块（扶抑为主 + 调候修正）

### 为什么不交给 LLM

同一个盘今天喜火明天喜水，用户立刻不信你。高频入口（每日运势）的喜忌必须 100% 确定性。

### 规则定义

**主规则：扶抑法**
- 日主强弱打分 = `得令权重 + 得地权重 + 得势权重`（具体权重待 spike 标定）
- 身强 → 用克泄耗（官杀、食伤、财）
- 身弱 → 用生扶（印、比劫）
- 输出固定的喜用五行 + 忌五行

**修正规则：调候法**
- 触发条件：生在极寒月（子、丑）或极热月（午、未）
- 叠加调候用神：冬生喜火暖局、夏生喜水润局
- **优先级高于普通扶抑**——调候用神与扶抑用神冲突时，取调候

**输出结构**
```json
{
  "day_master_strength": "strong" | "weak" | "balanced",
  "favorable_elements": ["火", "土"],
  "unfavorable_elements": ["水", "金"],
  "tiaoshou_applied": true,
  "method": "扶抑+调候"
}
```

输出**存进 ChartSnapshot**，LLM 只负责把这个结构润色成命书话术，不负责判断。

### 覆盖范围与边界

- 覆盖约 90% 的命盘（普通旺衰盘）
- **不覆盖**：从格、专旺（曲直/炎上/稼穑/从革/润下）、化气格——这些是"格局"范畴，已 MVP 砍掉
- 边界 case（旺衰接近平衡）：输出 `"day_master_strength": "balanced"`，LLM 话术改写为"日主中和，喜忌随大运流转"

### 从格边界处理（决策 1b，D3 锁定 2026-07-10）

10% 的从格/专旺命盘不能硬塞扶抑（会得反结果），也不能反转扶抑（又掉进决策 5 砍掉的格局坑）。**用 3-4 条 if 检测从格特征，命中则诚实降级**：

**检测规则**（不是完整格局判定，只是简单阈值）：
- **专旺格**：本命四柱 8 字中，日主同气五行 ≥ 6 个
- **从格**：日主孤立（四柱天干无印/比劫同气）+ 某一行 ≥ 5 个

**命中处理**：
```json
{
  "day_master_strength": "special_pattern",
  "pattern_hint": "zhuanwang" | "cong" | null,
  "favorable_elements": [],   // 留空，不输出错误结论
  "unfavorable_elements": [],
  "tiaoshou_applied": false,
  "method": "扶抑+调候（从格特征检测命中，未判定具体格局）"
}
```

**LLM prompt 改写**：命中 `special_pattern` 时，prompt 追加约束："此命局呈现专旺/从格特征，传统格局判定各派不同，以下解读仅供参考，不下硬性喜忌结论"。

**理由**：检测规则简单（不陷入流派争议），LLM 只做"告知"不做"判定"，与决策 1（喜忌确定性）+ 决策 5（格局砍掉）哲学一致。10% 用户拿到诚实的"需人工解读"标记，比拿到错的喜忌强。

### 与"格局砍掉"的关系

格局判定 MVP 砍掉不变（详见 `bazi-app-design-doc.md` 排盘规则 §4）。喜忌用扶抑+调候**不是格局判定**——只判日主旺衰方向 + 极端月份调候，不下"正官格 / 偏印格"等硬分类。两者并行不悖。

## 决策 2：神煞 = 固定 20 个清单，《三命通会》单一来源

### 选定清单（20 个，吉凶各半）

**吉神（10）**：天乙贵人、太极贵人、文昌、天德、月德、驿马、桃花（咸池）、将星、华盖、金舆、禄神

**凶煞（9）**：羊刃、劫煞、亡神、孤辰、寡宿、元辰、灾煞、天罗地网、红艳

（吉神 11 + 凶煞 9 = 20；红艳在传统中吉凶参半，归凶煞以平衡）

> 注：清单实际数 20，请实现时按 20 个做。

### 单一来源原则

**所有神煞的起法统一以《三命通会》为准**，一个来源查到底。各家起法冲突是神煞系统最大的坑，决不东拼西凑。

实现形态：静态查表（`SHENSHA_TABLE`），输入四柱直接查，确定性输出。

### 输出结构
```json
{
  "shensha": [
    { "name": "天乙贵人", "position": "日柱", "source": "三命通会" },
    { "name": "文昌", "position": "时柱", "source": "三命通会" },
    ...
  ]
}
```

### 为什么是 20 个

- 太少（< 10）：命书内容单薄，LLM 没素材
- 太多（> 30）：LLM 内容发散，命中率虚高
- 20 个左右：足够撑起命书丰富度，又控制住 LLM 发散

## 决策 3：ChartSnapshot = 不可变值对象 + 内容寻址

### 核心规则

**1. 唯一性（内容寻址 ID）**

```python
snapshot_id = sha256(
    "阳历出生时刻（精确到时辰, 2小时粒度）" +
    "性别" +
    "出生地经度（用于真太阳时）" +
    "子时规则（zi_next_day | zero_oclock）"
).hexdigest()
```

输入一样就是同一个 snapshot，天然去重、天然可缓存。

**2. 修改 = 新建（不可变）**

用户改生日 / 切时辰 / 换城市 / 切子时规则 → 不改旧对象，生成或指向一个新 snapshot。旧的保留，基于旧 snapshot 的命书、运势、合盘记录仍然有效、不被污染。

**3. 多人命盘（归属关系解耦）**

- 一个用户可挂多个 snapshot（自己 + 家人 + 伴侣）
- 归属关系放在关联表 `user_snapshots(user_id, snapshot_id, alias)`，snapshot 本身**不含归属信息**
- 两个用户生日相同 → 共享同一个 snapshot、共享缓存

**4. 合盘 = 两个 snapshot_id 的有序对**

```python
compatibility_id = sha256(
    min(snapshot_a_id, snapshot_b_id) + "|" +
    max(snapshot_a_id, snapshot_b_id) + "|" +
    context  # general | marriage | business
).hexdigest()
```

合盘结果也可按这对 id + context 做内容寻址缓存。生日相同的两对用户合盘 → 共享同一个合盘缓存。

### 数据模型影响（修订 `bazi-app-design-doc.md` §Data Model）

原设计的 `ChartSnapshot` SwiftData 类需要重构：

```swift
@Model
class ChartSnapshot {
    @Attribute(.unique) var contentHash: String  // 内容寻址 ID
    var birthSolarTime: Date       // 阳历出生时刻（精确到时辰）
    var gender: String
    var cityLongitude: Double      // 出生地经度（不再存 cityKey，直接存经度）
    var ziHourRule: String
    
    // 排盘结果（结构化 JSON）
    var pillars: Data
    var tenGods: Data
    var nayin: Data
    var shensha: [String]
    var elementBalance: [String: Int]
    var luckPillars: Data
    
    // 决策 1 新增：确定性喜忌
    var dayMasterStrength: String  // strong | weak | balanced
    var favorableElements: [String]
    var unfavorableElements: [String]
    var tiaoshouApplied: Bool
    
    // 计算规则快照（library 版本、offset、calculated_at）
    var calcRuleSnapshot: Data
    
    // 不含：user 关系、alias、修改历史
}

@Model
class UserSnapshotLink {
    @Attribute(.unique) var id: UUID
    var userId: String          // 本地用户标识（v1 单用户可以是固定值）
    var snapshotHash: String    // FK → ChartSnapshot.contentHash
    var alias: String           // "我自己" | "妈妈" | "男友" 等
    var createdAt: Date
}
```

**关键差异**：
- `id` 不再是 UUID，是 `contentHash`（内容寻址）
- 移除 `cityKey`，直接存 `cityLongitude`（snapshot 不依赖城市名，只依赖经度）
- 新增喜忌字段（决策 1 输出）
- 用户归属解耦到 `UserSnapshotLink`

### Schema 演化策略（决策 3b，D1 锁定 2026-07-10）

ChartSnapshot 是不可变内容寻址 + SwiftData 要求 schema 可演化，两者冲突。**采用 B+C 融合方案：身份 hash 与结构版本分离**。

**核心原则**：
- `contentHash` 永远按出生信息算（**不含** schema_version），保证多用户共享缓存命中率最大
- `schema_version` 是独立字段（Int，从 1 递增），追踪数据结构版本
- 易变复杂结构（pillars/十神/纳音/神煞/喜忌/luck_pillars）走 JSON `payload` Data 字段，SwiftData 核心 schema 几乎不变
- 加字段 = `payload` JSON 加 key + `schema_version +1`，老 snapshot lazy 重算（按需触发，启动时不批量）

**修订后的 SwiftData 模型**：

```swift
@Model
class ChartSnapshot {
    @Attribute(.unique) var contentHash: String  // SHA256(birth + gender + lon + rule)
    var schemaVersion: Int = 1                   // 数据结构版本，独立于 contentHash
    var birthSolarTime: Date
    var gender: String
    var cityLongitude: Double
    var ziHourRule: String

    var calcRuleSnapshot: Data  // JSON: library 版本、sect、offset、calculated_at
    var payload: Data           // JSON: pillars/十神/纳音/神煞/喜忌/luck_pillars 等易变结构

    // 不含：user 关系、alias、修改历史
}
```

**演化场景**：
| 场景 | 动作 |
|---|---|
| 加新字段（如未来加格局字段） | `payload` JSON 加 key + schemaVersion +1，老 snapshot lazy 重算填充 |
| 改字段类型 | schemaVersion +1 + 一次性 migration script 遍历重算 |
| lunar_python 升级 | `calcRuleSnapshot.library` 版本字段变化，按需重算（影响精度，触发提示） |

**rejected alternatives**：
- Hash 嵌 schema_version：跨用户共享缓存命中率断崖
- 纯 SwiftData 原生 migration：snapshot 不再严格不可变，复杂嵌套结构 migration 易错
- 完全 JSON 兜底（无 SwiftData 字段）：失去 query 能力

## 决策 4：AI 解读缓存层 = 客户端 + 后端 SQLite（D2 锁定 2026-07-10）

### 问题

AI 解读（深度解析/合盘/每日运势）是项目最大的 AI provider 成本来源。每次调用 200-700 字输出，按 token 算钱。如果不缓存：
- 同用户重复打开 = 重复调上游模型（钱白烧）
- 多用户同出生信息 = 各自调上游模型（content_hash 内容寻址的跨用户共享价值失效）
- 用户每次等 3-5 秒（UX 差）

### 两级缓存设计

```
客户端 SwiftData                         后端 SQLite
┌────────────────────────┐               ┌──────────────────────────┐
│ InterpretationCache    │  miss → API → │ interpretation_cache     │
│ content_hash / module  │               │ content_hash / module    │
│ prompt_version / date  │               │ prompt_version / date    │
│ provider / model       │               │ prompt_hash              │
│ interpretation         │ ← 命中返回 ── │ provider / model         │
│ generated_at           │               │ interpretation / time    │
└────────────────────────┘               └──────────────────────────┘
                                                ↑ miss → AIClient
                                                  ├─ Anthropic
                                                  └─ OpenAI
```

**客户端缓存（SwiftData）**：每用户设备缓存自己的解读，命中零延迟
**后端缓存（SQLite）**：跨用户共享，同出生信息的两用户命中同一缓存（content_hash 内容寻址最大价值）

### 缓存键设计

```python
# 深度解析
cache_key = (chart_hash, "bazi_deep", prompt_version, None, provider, model)

# 合盘
cache_key = (compatibility_hash, "compatibility", prompt_version, None, provider, model)
# compatibility_hash = SHA(a_hash, b_hash, context)

# 每日运势（target_date 是动态维度，不能省）
cache_key = (chart_hash, "daily_fortune", prompt_version, target_date, provider, model)
```

**`prompt_version` 必须独立字段**：prompt 改了，所有旧缓存失效（desired behavior，因为 prompt 改了输出也变）。

**`provider + model` 必须进入两级缓存身份**：切换 Anthropic/OpenAI 或同一家更换模型时，旧结果不能冒充当前模型的缓存。后端 SQLite 主键额外包含渲染后 prompt 的 `prompt_hash`，防止相同业务 hash 携带不同 context 时互相污染。

**客户端身份解析**：三个 orchestrator 在读 SwiftData AI 缓存前先调用 `GET /api/health`，请求使用 `reloadIgnoringLocalCacheData`，只读取 provider/model 完全匹配的行。health 失败、身份畸形或缓存读写失败都进入现有 AI error 状态，不以未知身份缓存伪装成功。`POST /api/interpret` 响应中的 provider/model 是最终生成身份；部署竞态导致它与此前 health 不同时，以 interpret 响应为准写入缓存。

**迁移与历史文本**：`InterpretationCache.provider/model` 采用 optional 以承接旧 SwiftData 行，但 nil 旧行永不命中新查询，新写入必须有完整身份。每日运势和合盘 snapshot 的来源字段同样 optional；旧文本仍可回看，但不能阻止当前 provider 发起新调用，也不能显示为当前缓存命中。

### Singleflight（暂不做）

side project 阶段用户数少，并发同 hash 概率低。singleflight（asyncio.Lock + 状态机，合并同时请求为一个 provider 调用）留到 v2 加。**不做 Redis**——SQLite 文件级足够，Redis 运维成本不该烧。

### 失效策略

| 场景 | 动作 |
|---|---|
| prompt 改了 | `prompt_version +1`，老缓存自然失效（不主动删，新 key 不会命中老缓存） |
| provider 或 model 切换 | 新身份自然 miss，不读取另一身份缓存；不自动 fallback |
| lunar_python 升级（精度变化） | `calcRuleSnapshot.library` 版本变化 → chart_hash 不变（出生信息没变），但建议清空缓存（prompt_version +1） |
| 喜忌规则引擎更新 | 触发 lazy 重算所有 snapshot，prompt_version +1 |

### 实现位置

- 后端：启动时 factory 选择 `AIClient`，FastAPI 路由按完整缓存键查 SQLite，miss 后调用当前 client
- 客户端：集中式 `AIIdentityResolver` 先解析 health 身份，再按完整身份查本地；miss 走 API

### Provider 选择与故障策略

- `AI_PROVIDER=anthropic|openai`，默认 `anthropic`；非法值启动失败
- Anthropic 默认 `claude-sonnet-4-6`，OpenAI 默认 `gpt-5.5`
- OpenAI 走官方 Responses API，Anthropic 保持 Messages API；统一 15 秒超时和 1024 输出 token
- 选择的 provider 缺 key 时非 AI 路由仍能启动，`/api/interpret` 返回 503
- 上游超时、网络错误、拒绝、空/畸形响应和非成功状态都显式传播；禁止调用另一供应商兜底
