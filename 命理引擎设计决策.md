## 设计哲学

整个命理引擎遵循一条主线：

> **排盘输入 → 内容寻址生成不可变 ChartSnapshot → 确定性算好喜忌和神煞 → LLM 只做润色不做判断**

这样命理引擎是确定的、可缓存的、可复现的；LLM 只在最外层负责"说人话"。所有可能有流派争议的判断（用神、格局分类）要么规则定死（喜忌用扶抑+调候），要么砍掉（格局判定），绝不交给 LLM 现场判。

## 决策 1：喜忌 = 确定性模块（扶抑为主 + 调候修正）

### 为什么不交给 LLM

同一个盘今天喜火明天喜水，用户立刻不信你。高频入口（每日运势）的喜忌必须 100% 确定性。

### 规则定义

**主规则：扶抑法**
- 日主强弱打分 = `得令权重 + 得地权重 + 得势权重`（具体权重待 spike 标定）
- 身强 → 用克泄耗（官杀、食伤、财）
- 身弱 → 用生扶（印、比劫）
- 输出固定的喜用五行 + 忌五行

**修正规则：调候法**
- 触发条件：生在极寒月（子、丑）或极热月（午、未）
- 叠加调候用神：冬生喜火暖局、夏生喜水润局
- **优先级高于普通扶抑**——调候用神与扶抑用神冲突时，取调候

**输出结构**
```json
{
  "day_master_strength": "strong" | "weak" | "balanced",
  "favorable_elements": ["火", "土"],
  "unfavorable_elements": ["水", "金"],
  "tiaoshou_applied": true,
  "method": "扶抑+调候"
}
```

输出**存进 ChartSnapshot**，LLM 只负责把这个结构润色成命书话术，不负责判断。

### 覆盖范围与边界

- 覆盖约 90% 的命盘（普通旺衰盘）
- **不覆盖**：从格、专旺（曲直/炎上/稼穑/从革/润下）、化气格——这些是"格局"范畴，已 MVP 砍掉
- 边界 case（旺衰接近平衡）：输出 `"day_master_strength": "balanced"`，LLM 话术改写为"日主中和，喜忌随大运流转"

### 与"格局砍掉"的关系

格局判定 MVP 砍掉不变（详见 `bazi-app-design-doc.md` 排盘规则 §4）。喜忌用扶抑+调候**不是格局判定**——只判日主旺衰方向 + 极端月份调候，不下"正官格 / 偏印格"等硬分类。两者并行不悖。

## 决策 2：神煞 = 固定 20 个清单，《三命通会》单一来源

### 选定清单（20 个，吉凶各半）

**吉神（10）**：天乙贵人、太极贵人、文昌、天德、月德、驿马、桃花（咸池）、将星、华盖、金舆、禄神

**凶煞（9）**：羊刃、劫煞、亡神、孤辰、寡宿、元辰、灾煞、天罗地网、红艳

（吉神 11 + 凶煞 9 = 20；红艳在传统中吉凶参半，归凶煞以平衡）

> 注：清单实际数 20，请实现时按 20 个做。

### 单一来源原则

**所有神煞的起法统一以《三命通会》为准**，一个来源查到底。各家起法冲突是神煞系统最大的坑，决不东拼西凑。

实现形态：静态查表（`SHENSHA_TABLE`），输入四柱直接查，确定性输出。

### 输出结构
```json
{
  "shensha": [
    { "name": "天乙贵人", "position": "日柱", "source": "三命通会" },
    { "name": "文昌", "position": "时柱", "source": "三命通会" },
    ...
  ]
}
```

### 为什么是 20 个

- 太少（< 10）：命书内容单薄，LLM 没素材
- 太多（> 30）：LLM 内容发散，命中率虚高
- 20 个左右：足够撑起命书丰富度，又控制住 LLM 发散

## 决策 3：ChartSnapshot = 不可变值对象 + 内容寻址

### 核心规则

**1. 唯一性（内容寻址 ID）**

```python
snapshot_id = sha256(
    "阳历出生时刻（精确到时辰, 2小时粒度）" +
    "性别" +
    "出生地经度（用于真太阳时）" +
    "子时规则（zi_next_day | zero_oclock）"
).hexdigest()
```

输入一样就是同一个 snapshot，天然去重、天然可缓存。

**2. 修改 = 新建（不可变）**

用户改生日 / 切时辰 / 换城市 / 切子时规则 → 不改旧对象，生成或指向一个新 snapshot。旧的保留，基于旧 snapshot 的命书、运势、合盘记录仍然有效、不被污染。

**3. 多人命盘（归属关系解耦）**

- 一个用户可挂多个 snapshot（自己 + 家人 + 伴侣）
- 归属关系放在关联表 `user_snapshots(user_id, snapshot_id, alias)`，snapshot 本身**不含归属信息**
- 两个用户生日相同 → 共享同一个 snapshot、共享缓存

**4. 合盘 = 两个 snapshot_id 的有序对**

```python
compatibility_id = sha256(
    min(snapshot_a_id, snapshot_b_id) + "|" +
    max(snapshot_a_id, snapshot_b_id) + "|" +
    context  # general | marriage | business
).hexdigest()
```

合盘结果也可按这对 id + context 做内容寻址缓存。生日相同的两对用户合盘 → 共享同一个合盘缓存。

### 数据模型影响（修订 `bazi-app-design-doc.md` §Data Model）

原设计的 `ChartSnapshot` SwiftData 类需要重构：

```swift
@Model
class ChartSnapshot {
    @Attribute(.unique) var contentHash: String  // 内容寻址 ID
    var birthSolarTime: Date       // 阳历出生时刻（精确到时辰）
    var gender: String
    var cityLongitude: Double      // 出生地经度（不再存 cityKey，直接存经度）
    var ziHourRule: String
    
    // 排盘结果（结构化 JSON）
    var pillars: Data
    var tenGods: Data
    var nayin: Data
    var shensha: [String]
    var elementBalance: [String: Int]
    var luckPillars: Data
    
    // 决策 1 新增：确定性喜忌
    var dayMasterStrength: String  // strong | weak | balanced
    var favorableElements: [String]
    var unfavorableElements: [String]
    var tiaoshouApplied: Bool
    
    // 计算规则快照（library 版本、offset、calculated_at）
    var calcRuleSnapshot: Data
    
    // 不含：user 关系、alias、修改历史
}

@Model
class UserSnapshotLink {
    @Attribute(.unique) var id: UUID
    var userId: String          // 本地用户标识（v1 单用户可以是固定值）
    var snapshotHash: String    // FK → ChartSnapshot.contentHash
    var alias: String           // "我自己" | "妈妈" | "男友" 等
    var createdAt: Date
}
```

**关键差异**：
- `id` 不再是 UUID，是 `contentHash`（内容寻址）
- 移除 `cityKey`，直接存 `cityLongitude`（snapshot 不依赖城市名，只依赖经度）
- 新增喜忌字段（决策 1 输出）
- 用户归属解耦到 `UserSnapshotLink`

## 实现优先级

```
P0  后端排盘基础（lunar_python spike → 真实字段映射）
P0  喜忌规则引擎（扶抑+调候，~30 个测试用例）
P0  神煞查表（《三命通会》20 个）
P1  ChartSnapshot 内容寻址 + SwiftData 模型
P1  UserSnapshotLink + 多人命盘 UI
P2  合盘内容寻址缓存
```
