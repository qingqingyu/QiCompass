## Problem Statement

Build a native iOS app that brings traditional Chinese divination (BaZi birth charts, LiuYao I Ching hexagrams, Temple Fortune Sticks) to life through AI-powered personalized interpretations. The app targets overseas Chinese (25-45) and Westerners interested in Eastern culture (18-35). The core differentiator: a unified fate map timeline (卷轴) that connects all divination results into a personal divination autobiography — not three isolated tools, but three resolutions of the same system.

## What Makes This Cool

The scroll timeline IS the product. Every reading — a BaZi chart, a LiuYao hexagram, a fortune stick pull — becomes a pin on a personal scroll. Over time, users build a visual autobiography told through divination. The "palace antique" aesthetic (deep ink-black, gold accents, classical Chinese typography) makes it feel like a dimly-lit temple at dusk, not a generic fortune-telling app. The coin-casting ritual with haptics, the brush-stroke text reveal, the incense-smoke transitions — these are the full product vision. V1 ships the gold-on-dark foundation and simple transitions; the ritual polish iterates post-launch.

## Constraints

- Native iOS (Swift/SwiftUI) — no cross-platform frameworks
- Backend proxy for Claude API calls (user already has this)
- BaZi calculation must use established libraries (cnlunar/sxtwl), not hand-rolled algorithms
- Three modules in v1: BaZi, LiuYao, Fortune Sticks
- Side project — built by one person in spare time
- Palace antique aesthetic: #0d0b08 background, #c9a03c gold, #f5d785 bright gold, #e8dcc8 text

## Premises

1. **Native iOS is the right platform.** The target audience (overseas Chinese, spiritual-curious Westerners) discover apps through the App Store. Haptics (UIImpactFeedbackGenerator for coin flips) are core to the experience.
2. **Backend proxy for API security.** No API keys in the client. The user already has a proxy infrastructure.
3. **Library-based BaZi calculation.** Using cnlunar for base solar-to-lunar conversion and sxtwl for solar-term-accurate month pillar calculation on the backend. Hand-rolling this algorithm (as in the original PRD) risks accuracy edge cases. Key concern: Zi hour boundary (子时, the debate over whether the Zi hour starts at 23:00 or 00:00 affects day pillar assignment). The chosen library must handle this correctly.
4. **Three modules connected by a fate map timeline.** The scroll timeline is the main screen. Modules are accessed through "+" entry points on the timeline. Each reading drops as a card on the scroll, creating a personal divination journal over time.
5. **Feeling over features.** The palace antique aesthetic, animations, haptics, and typography aren't decoration — they're the product. Users come for the accuracy, they stay for the atmosphere.

## Chosen Approach

**Approach C: Fate Map Timeline.** The user chose this over the recommended Approach B. The fate map is the ambitious version — it turns three separate divination tools into a single coherent product. The scroll timeline gives users a reason to come back (their personal divination history becomes richer over time) and a reason to share (the scroll is a visual artifact, not just text). Risk is manageable because the underlying modules are the same as Approach B; the timeline is an additional layer on top.

### Module Specifications

#### BaZi (八字命理)
- **Input:** Birth date (DatePicker), birth time (DatePicker with hour precision), optional name
- **Calculation:** Backend endpoint. App sends birth datetime as UTC + timezone offset (e.g., `1990-03-15T14:30:00+08:00`). Backend uses cnlunar for base solar-to-lunar calendar conversion, sxtwl for solar-term-accurate month pillar boundaries. Returns four pillars, five-element balance, luck pillars (大运), current luck pillar phase
- **Display:** 4-column layout (year, month, day, hour), each column shows heavenly stem on top / earthly branch below, with five-element color borders. Five-element statistics badges. Current luck pillar band
- **AI interpretation:** Claude receives structured chart data (not raw user input) and generates ~500-word analysis of elemental balance, day master traits, pattern recognition, personality, and luck pillar guidance
- **Fate map integration:** BaZi reading is the first card on the timeline and also populates the luck pillar top band (cached for offline display). The BaZi card scrolls with the timeline (not pinned). If user has not completed a BaZi reading, the top band shows a prompt card: "Complete your BaZi birth chart to unlock your luck pillar timeline" with a CTA button

#### LiuYao (六爻占卜)
- **Input:** Optional question text
- **Interaction:** Six rounds of coin-flipping. Each round: user taps to flip three virtual coins with haptic feedback (UIImpactFeedbackGenerator). Animation: coin rotation + fade. Running tally builds hexagram from bottom to top
- **Calculation:** Each coin: heads=3, tails=2. Three coins summed: 6=老阴 (changing yin), 7=少阳 (young yang), 8=少阴 (young yin), 9=老阳 (changing yang). Six yao are generated sequentially from bottom to top (yaoValues[0] = 初爻 bottom line, yaoValues[5] = 上爻 top line). Binary encoding: yin=0, yang=1, with yaoValues[0] as least significant bit. Maps to 64 hexagrams via local JSON lookup table (King Wen sequence, bundled in app assets). Changing lines (6 and 9) identified
- **Display:** Hexagram line diagram (solid/broken/changing) + hexagram name + English name + yao type badges
- **AI interpretation:** Claude receives hexagram number, name, changing line positions, and user question. ~400-word analysis of hexagram meaning, changing line text, and specific guidance
- **Fate map integration:** Each LiuYao reading becomes a "tactical pin" on the timeline, tagged with the question topic

#### Fortune Sticks (灵签)
- **Input:** Optional question text
- **Interaction:** Tap-and-hold with shake animation. Bamboo stick bundle animates (shake for ~1.5s). One stick "falls out" with a subtle sound. Device shake (CMMotionManager) deferred to post-launch iteration.
- **Calculation:** Random number in range `1..corpus_count` (not 1-100), so every drawn number has a poem. Stick text from curated JSON corpus (minimum 8 sticks at launch, expand to 100 over time)
- **Display:** Stick number (64px), traditional stick poem in card format with classical letter-spacing
- **AI interpretation:** Claude receives stick number, poem text, user question. ~350-word analysis of fortune level (上/中/下), poem interpretation, and practical advice
- **Fate map integration:** Each fortune stick pull becomes a "meditative snapshot" on the timeline

#### Fate Map Timeline (卷轴)
- **Main screen:** Scroll-style vertical timeline with side decoration lines simulating physical scroll edges
- **Top band:** Current luck pillar from BaZi (visible after BaZi reading completed; otherwise shows prompt card with CTA)
- **Timeline nodes:** Chronological reading cards with color-coded type badges (red=BaZi, blue=LiuYao, gold=Sticks)
- **Entry points:** Single "+" floating action button at bottom-right. Tap reveals bottom sheet with three options: 八字 / 六爻 / 灵签
- **Persistence:** SwiftData for local storage of all readings. Chosen for v1 simplicity. If query performance becomes an issue with large reading histories, migration path is to raw CoreData or SQLite via GRDB
- **Empty state:** When no readings exist, timeline shows a single card: "Begin your journey — cast your first divination" with module picker

#### Data Model

```swift
// SwiftData models (not protocols — SwiftData requires @Model classes)
@Model
class ReadingEntry {
    @Attribute(.unique) var id: UUID
    var timestamp: Date
    var type: ReadingType  // enum: .bazi, .liuyao, .fortuneStick
    var question: String?
    var interpretation: String
    // Module-specific JSON payload
    var moduleData: Data  // Encoded BaZiData / LiuYaoData / FortuneStickData

    // Computed: decode moduleData based on type
}

enum ReadingType: String, Codable {
    case bazi, liuyao, fortuneStick
}

// Module-specific data structs (Codable, stored as JSON in moduleData)
struct BaZiData: Codable {
    let name: String?
    let pillars: [Pillar]  // year, month, day, hour (4 items)
    let elementBalance: [String: Int]  // "wood": 2, "fire": 1, etc.
    let currentLuckPillar: String
    let currentYearPillar: String
    let luckPillarCached: Bool  // true = available offline
}

struct LiuYaoData: Codable {
    let hexagramNumber: Int
    let hexagramName: String
    let yaoValues: [Int]  // 6 values, each 6-9
    let changingLines: [Int]  // positions of changing yao
}

struct FortuneStickData: Codable {
    let stickNumber: Int
    let poemText: String
    let fortuneLevel: String?  // 上/中/下, from AI
}
```

Timeline entries are ordered by `timestamp` descending (newest first). BaZi readings are also cached separately for the luck pillar top band (available offline after first calculation). Type-specific rendering uses the `ReadingType` enum for badge colors and card layout. BaZi is limited to one reading per birth date; the most recent BaZi reading is used for the timeline top band. Schema migration uses SwiftData's `VersionedSchema` and `SchemaMigrationPlan` for versioned evolution.

#### Reading Counter

Daily AI reading limit (10/day) is persisted via UserDefaults with a date key. On each new day (midnight local time), the counter resets. No live countdown — the rate limit UI shows "今日机缘已尽，明日再来" (today's fortune is exhausted, return tomorrow) with the reset time.

#### Error Handling

All modules handle failure states consistently:
- **Network offline / API timeout (>15s):** Show ink-splash error card with "天意未明" (heaven's will unclear) message and retry button. Reading is NOT saved to timeline.
- **Malformed API response:** Same error card. Log the error locally for debugging.
- **BaZi calculation library error:** Return empty pillars with error message. User prompted to retry or verify birth time.
- **Rate limit exceeded:** Show "今日机缘已尽" (today's fortune is exhausted) card with a countdown to next available reading. Client-side enforcement: max 10 AI readings per day.

#### Accessibility

- **Reduced motion:** When iOS Reduce Motion is enabled, disable coin-flip and shake animations. Show instant results with fade transitions instead.
- **VoiceOver:** Add labels to all interactive elements (coin buttons, shake area, timeline cards). Reading interpretation is fully accessible via VoiceOver.
- **Contrast:** Gold (#f5d785) on dark (#0d0b08) passes WCAG AA (contrast ratio ~10:1). Body text (#e8dcc8) on dark passes AA (~11:1).
